# 6. Регламенты кодирования

## 6.1. Линтеры

Используемые инструменты применяются в рамках CI-пайплайнов.

### 6.1.1. Frontend

Для frontend-части используются следующие инструменты:

| Инструмент | Назначение |
|-----------|------------|
| **ESLint** | Проверка качества TypeScript/React-кода |
| **Prettier** | Единое форматирование кода |
| **TypeScript compiler (`tsc`)** | Проверка типов |
| **Vite build** | Проверка корректности сборки frontend-приложения |

Набор проверок ESLint:

- `eslint:recommended`;

- `@typescript-eslint/recommended`;

- `plugin:react/recommended`;

- `plugin:react-hooks/recommended`;

- `plugin:jsx-a11y/recommended`;

- `prettier`.

### 6.1.2. Backend

Для backend-части используются следующие инструменты:

| Инструмент                   | Назначение                            |
| ---------------------------- | ------------------------------------- |
| **dotnet format**            | Проверка форматирования C#-кода       |
| **.editorconfig**            | Единые правила форматирования и стиля |
| **Roslyn analyzers**         | Статический анализ C#-кода            |
| **Nullable Reference Types** | Контроль nullable-контрактов          |
| **dotnet build**             | Проверка компиляции                   |

## 6.2 Правила именования переменных

### 6.2.1. Общие правила

Для всего проекта:
- имена должны быть осмысленными;
- запрещены сокращения по типу `obj`, `tmp`, `val`, если из контекста не
очевидно их значение;
- булевые переменные начинаются с префикса `is`/`has`/`can`/`should`;
- названия коллекций в множественном числе;
- названия исходных файлов должны соответствовать основной сущности или
компоненту внутри файла.

### 6.2.2. Frontend

| Элемент                      | Формат             | Пример                         |
| ---------------------------- | ------------------ | ------------------------------ |
| React-компоненты             | `PascalCase`       | `ComicsCard`, `ReadingPage`    |
| TypeScript-типы и интерфейсы | `PascalCase`       | `ComicsDto`, `UserProfile`     |
| Переменные                   | `camelCase`        | `currentPage`, `selectedGenre` |
| Функции                      | `camelCase`        | `fetchComics`, `handleSubmit`  |
| React hooks                  | `useCamelCase`     | `useAuth`, `useComicsList`     |
| Константы                    | `UPPER_SNAKE_CASE` | `MAX_PAGE_SIZE`                |
| CSS Modules-классы           | `camelCase`        | `cardWrapper`, `submitButton`  |
| Файлы компонентов            | `PascalCase.tsx`   | `ComicsCard.tsx`               |
| Файлы hooks                  | `use*.ts`          | `useAuth.ts`                   |
| Файлы утилит                 | `camelCase.ts`     | `formatDate.ts`                |

### 6.2.3. Backend

| Элемент              | Формат                              | Пример                            |
| -------------------- | ----------------------------------- | --------------------------------- |
| Классы               | `PascalCase`                        | `ComicsService`, `AuthController` |
| Интерфейсы           | `IPascalCase`                       | `IComicsRepository`               |
| Методы               | `PascalCase`                        | `GetComicsAsync`                  |
| Свойства             | `PascalCase`                        | `AverageRating`                   |
| Локальные переменные | `camelCase`                         | `currentUser`, `comicsId`         |
| Приватные поля       | `_camelCase`                        | `_dbContext`, `_logger`           |
| Константы            | `PascalCase` или `UPPER_SNAKE_CASE` | `MaxPageSize`                     |
| DTO                  | `PascalCase + Dto`                  | `ComicsDto`, `CreateComicsDto`    |
| Request-модели       | `PascalCase + Request`              | `LoginRequest`                    |
| Response-модели      | `PascalCase + Response`             | `AuthResponse`                    |
| Entity-классы        | `PascalCase`                        | `Comics`, `ReadingHistory`        |
| EF Core migrations   | Осмысленное имя                     | `AddReadingHistoryTable`          |

## 6.3. Правила кодирования
Для всего проекта:

1. Код должен быть читаемым без необходимости разбирать внутренние детали реализации.
2. Один модуль, класс, компонент или функция должны иметь одну ответственность.
3. Не допускается дублирование бизнес-логики.
4. Не допускается хранение секретов в репозитории.
5. Все ошибки должны обрабатываться явно.
6. Публичные API-контракты должны быть стабильными и документированными.
7. Любое изменение модели данных должно сопровождаться миграцией.
8. Pull request не должен содержать неиспользуемый код, временные комментарии, console.log, debugger, закомментированные блоки старой реализации.
9. Код должен проходить локальные проверки до открытия pull request.

### 6.3.1. Frontend

Обязательные правила:

1. Все новые файлы пишутся на TypeScript.
2. Использование any запрещено, кроме случаев, явно обоснованных в комментарии.
3. Компоненты должны быть функциональными.
4. Бизнес-логика не должна находиться внутри JSX-разметки.
5. Работа с серверным состоянием должна выполняться через React Query.
6. API-запросы должны быть вынесены в отдельный слой.
7. Компоненты не должны напрямую знать детали HTTP-клиента.
8. Повторяемая логика должна выноситься в custom hooks.
9. Формы должны иметь валидацию.
10. Все пользовательские состояния загрузки и ошибок должны быть обработаны.

### 6.3.2. Backend

1. Контроллеры не должны содержать бизнес-логику.
2. Бизнес-логика размещается в Application-слое.
3. Доменные сущности размещаются в Domain-слое.
4. Работа с БД и внешними сервисами размещается в Infrastructure-слое.
5. Все публичные методы, выполняющие I/O, должны быть асинхронными.
6. Для async-методов используется суффикс Async.
7. Entity Framework Core-запросы должны быть оптимизированы и не должны приводить к N+1-проблеме.
8. Все входные данные должны валидироваться.
9. Ошибки должны возвращаться в едином формате.
10. Нельзя возвращать EF Core entity напрямую из API.
11. Для API используются DTO/Request/Response-модели.
12. Все изменения схемы БД оформляются через EF Core migrations.
